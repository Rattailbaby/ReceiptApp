## SECTION A — PARAMS REFERENCE

The cleanup flows pass state between screens through URL query params. There is no in-memory return stack.

### A.1 Param catalog

`id` (client-detail.tsx, line 16)
- Source: route `?id=...`
- Type: string in URL, coerced via `Number(id)` at line 32.
- Required: yes — the screen finds the client via `clients.find(c => c.id === Number(id))`.
- If missing, malformed, or no matching client: line 93 returns the "Client not found." fallback view with a Back button. App does not crash.
- If the value is not a number (e.g. `id=foo`): `Number('foo')` → `NaN`, `find` returns `undefined`, same fallback view.

`txId` (client-detail.tsx, line 16)
- Source: route `?txId=...`
- Used by `useEffect` at lines 68-74 (runs ONCE on mount, dependency array `[]`):
  ```
  if (txId) {
    const tx = transactions.find(t => String(t.id) === String(txId));
    if (tx) { setSelectedTx(tx); setIsEditing(false); }
  }
  ```
- Required: no — if absent, user lands on the client's transaction list with no sheet open.
- If `txId` does not match any transaction (e.g. deleted between Explore tap and arrival): `tx` is undefined, `selectedTx` stays `null`, no sheet opens, user sees the client transaction list. No error message.
- Comparison is string-based (`String(t.id) === String(txId)`), so type coercion is safe here.

`addTx` (client-detail.tsx, line 16)
- Source: route `?addTx=1`
- Used at line 69: `if (addTx === '1') setShowAddSheet(true);`
- Set by index.tsx line 111 in the Quick Add → Pick Client flow.
- Required: no.
- If anything other than the literal string `'1'`: ignored.

`source` (client-detail.tsx, line 16; explore.tsx, line 13)
- Source: route `?source=explore` or `?source=tax`
- Used in client-detail to gate cleanup redirects (lines 262, 294, 315, 773, 1025).
- Used in explore.tsx (line 17) to install a hardware-back interceptor that returns to `/tax`.
- Required: no — when missing, normal router.back() behavior takes over.
- If misspelled (`source=Explore` etc.): treated as missing. None of the redirect logic fires.

`returnFilter` (client-detail.tsx, line 16)
- Source: route `?returnFilter=untagged` (or `missing`, or `flagged`)
- Required only when `source === 'explore'`. Without it, redirect logic at lines 262, 294, 315, 1025 short-circuits and the user gets normal back navigation.
- Special-case at line 773: Mark Reviewed only redirects if `returnFilter === 'flagged'`. Other values are ignored even if present.
- Drives the destination: `router.replace('/explore?filter=' + returnFilter)`. If `returnFilter` is an unknown value, Explore loads with no recognized filter → falls through to default "first 10 transactions" view (explore.tsx line 54).

`filter` (explore.tsx, line 13)
- Source: route `?filter=untagged` (or `missing`, or `flagged`)
- Drives the visible filter and empty-state copy (lines 35-41, 76-82).
- Required: no — without it, Explore shows the first 10 transactions as a generic search landing page.
- If unknown value (e.g. `?filter=banana`): falls through to `safeTransactions` (line 41), so the user sees ALL transactions. The filter label at line 33 will be `null`.

### A.2 Param consequence summary

| Scenario                                                          | Result                                                              |
|-------------------------------------------------------------------|---------------------------------------------------------------------|
| `/client-detail` (no params)                                      | "Client not found." fallback                                        |
| `/client-detail?id=999` (no such client)                          | "Client not found." fallback                                        |
| `/client-detail?id=1&txId=999` (tx missing or wrong client)       | Client list shown, no sheet                                         |
| `/client-detail?id=1&txId=5&source=explore` (no returnFilter)     | Sheet opens, but redirect logic disabled — Close goes via router.back() not router.replace |
| `/client-detail?id=1&txId=5&source=explore&returnFilter=banana`   | Sheet opens; redirect goes to `/explore?filter=banana` → all txs shown, no filter label |
| `/explore` (no params)                                            | Generic search landing, first 10 txs                                |
| `/explore?filter=banana`                                          | All txs, no filter label                                            |
| `/explore?filter=missing&source=tax` then Android back            | Hardware back → `router.replace('/tax')` (interceptor at line 18)   |

## SECTION B — ENTRY POINTS

There are five distinct entry points into the cleanup flow. Each puts the user at a different starting position.

### B.1 Home → Needs Attention → Explore (filtered list)

File: app/(tabs)/index.tsx, lines 174-211

Counts shown on the home screen "NEEDS ATTENTION" panel:
- `flaggedCount = safeTx.filter(t => t.flagged).length` (line 174)
- `missingReceiptCount = safeTx.filter(t => !t.receiptUri && !t.note).length` (line 175)
- `untaggedCount = safeTx.filter(t => !t.tagIds?.length).length` (line 176)

**INCONSISTENCY (potential blocker — flag for follow-up):**
The home `missingReceiptCount` requires `!t.note` AS WELL AS `!t.receiptUri`. The Explore filter (`filter === 'missing'`, line 38 of explore.tsx) only checks `!t.receiptUri`. This means:
- A transaction with no receipt but a note set is COUNTED in Explore's missing list.
- The same transaction is NOT COUNTED in the Home banner.
- Possible result: Home shows "0 missing receipts" or hides the row entirely (panel hidden when all three counts are 0, line 200), but Explore's missing list still has items.

Tap row → `router.push(\`/explore?filter=${item.filter}\`)` (line 204). Note: no `source` param is set for this entry. The Tax-back interceptor in explore.tsx is therefore inert.

### B.2 Tax → Missing Receipts card → Explore

File: app/(tabs)/tax.tsx, line 87

Tap → `router.push('/explore?filter=missing&source=tax')`.

This is the ONLY entry that sets `source=tax`. It causes explore.tsx to install a hardware-back interceptor that returns to `/tax` (lines 16-23).

**INCONSISTENCY (data, not a crash):**
The card displays:
- `totalMissing = totalLogged - totalWithReceipts` where `totalWithReceipts` filters by `t.receipt` boolean (line 25).
- But `missingCount = yearTxs.filter(t => !t.receiptUri).length` (line 27) uses `receiptUri`.
- Per CLAUDE.md, `receiptUri` is the source of truth. The dollar amount on this card is computed from a stale boolean. If a transaction has `receipt: true` but no `receiptUri`, it counts as "with receipt" for the dollar total but as "missing" by the count.
- This affects the headline "$X.XX missing" number, not the navigation behavior.

### B.3 Explore tab directly → no filter

User taps the "Search" tab in the bottom bar.
- `filter` param not present.
- Empty-state copy: "No transactions yet." (line 82) when no query and no filter.
- Otherwise shows first 10 transactions (line 54).
- This is search mode, not cleanup mode. Tapping a row navigates to client-detail WITHOUT `source` or `returnFilter` (line 88: `${filter ? '...' : ''}` — when filter is empty, no source is appended).

### B.4 Quick Add (Home FAB) → Pick Client → client-detail

Home FAB (index.tsx line 300) → `setShowQuickAdd(true)` modal → tap a client row → `handleQuickAddSelect` (line 108) → `router.push(\`/client-detail?id=${client.id}&addTx=1\`)`.

Not a cleanup entry, but listed because it touches client-detail with `addTx=1` and matters for back-button testing.

### B.5 Client tile / list row on Home → client-detail (no cleanup context)

index.tsx lines 234, 255 — direct push to `/client-detail?id=${client.id}` with no other params.

Not a cleanup entry, but transactions opened from here will not show cleanup buttons because `source !== 'explore'`.

If empty, save calls `editTransaction(id, { tagIds: [] })` — no behavior change but a write to AsyncStorage. Redirect condition at line 1026 sees `editTagIds.length > 0` is false → no redirect.

### E.2 Buttons shown in detail sheet (source=explore, returnFilter=flagged)

- The block at line 784 (Add Tags / Add Receipt) is NOT rendered for `returnFilter === 'flagged'`. So if a flagged tx ALSO lacks tags or a receipt, the user can:

  - Use the "Edit" button (line 856) to open the edit form, then add tags via the picker there.
  - But there is no first-class "Add Tags" affordance like the other flows offer.

### E.3 Return behavior after fix

|-------------------------------------------------|------------------------------------------------------------------------|

If the user then uses the lower Add Receipt or the Edit form to fix the remaining issues:

### E.4 Edge cases

- **Transaction is flagged but already has tags + receipt (the clean case)**: Mark Reviewed → instant redirect. Empty filter list possible if it was the last flagged item.
- **User clears flag, then sees no remaining issues, but doesn't tap anything else**: The Mark Reviewed handler already redirected at line 776 because `!txNeedsCleanup` was true. They never get a chance to "do nothing" — the redirect is automatic.
- **User clears flag, sheet stays open due to other issues, then closes via the Close button**: `closeTxDetail` (line 259) redirects unconditionally (no `txNeedsCleanup` check). User exits to `/explore?filter=flagged` and the tx is now NOT in the flagged list (flag is cleared). No bug, just worth noting that "close" silently completes the flow.
- **Re-flag the same transaction during the session**: Mark Reviewed only sets to false. There's no "re-flag" affordance in this sheet. Not in scope.

## SECTION F — FLOW: MULTIPLE ISSUES (missing tags AND receipt)

This is not a separate filter — it's the worst-case overlap of flows C and D.

### F.1 Path

The flow followed depends on which filter the user entered through.

Entered via UNTAGGED:
- Detail sheet shows Add Tags primary, Add Receipt ghost (opacity 0.6 because both missing — line 789).
- Fix order: Add Tags → tags saved → sheet stays open. Add Receipt becomes primary. → Add Receipt → redirect.

Entered via MISSING:
- Detail sheet shows Add Receipt primary, Add Tags ghost (opacity 0.6 — line 799).
- Fix order: Add Receipt → photo → sheet stays open. Add Tags becomes primary. → Add Tags → redirect.

Entered via FLAGGED with both also missing:
- Only Mark Reviewed visible at top. Lower "📷 Add Receipt" visible. No inline Add Tags.
- See Section E.

### F.2 Behavior matrix — multi-issue resolution

| Path through fixes (entered via untagged)                       | What happens                                                |
|-----------------------------------------------------------------|-------------------------------------------------------------|
| Add Tags → Save → still missing receipt                         | Tag modal closes. Sheet open. Add Tags now greyed. Add Receipt now orange primary. |
| Add Receipt → photo → still missing tags                        | Camera closes. Sheet open. Add Receipt now greyed (opacity 0.3). Add Tags primary. |

### F.3 Edge cases

- **In-session timing**: between tapping "Add Tags" and the modal opening, the underlying transactions array is unchanged. `selectedTx` snapshot is used. No race that I can construct.

## SECTION G — UNIVERSAL EDGE CASES

These apply to all four flows (untagged, missing, flagged, multi-issue) regardless of entry point.

### G.1 Transaction deleted mid-cleanup

**By the user, from the same sheet** (tap trash icon, confirm):
- `deleteSingleTx` (line 368) calls `deleteTransaction` then `closeTxDetail`.
- `closeTxDetail` redirects to `/explore?filter=...` if source/returnFilter set. Tx is gone from store. List rebuilds without it.
- Verified safe.

**By another mechanism** (impossible in current single-user app, but if it happened):
- `selectedTx` state holds the stale snapshot. UI shows the deleted tx. Save/edit calls would target an id that no longer exists in the store.
- The store handlers (`editTransaction`, `deleteTransaction`) operate on arrays via `.map`/`.filter`. If the id is missing, they no-op silently — no crash.

### G.2 Client deleted mid-cleanup

While the user is on `/client-detail?id=X&txId=Y`:
- The store's `deleteClient` (not in these files but visible from index.tsx line 152) removes the client.
- On next render of client-detail, `client = clients.find(c => c.id === Number(id))` returns undefined.
- Line 93 short-circuits to "Client not found." view with a Back button.
- The cleanup sheet, if open, vanishes (since the entire screen is replaced by the fallback view).
- Hardware back from this fallback follows system default.

This case is reachable from multi-tab use (user on a different tab mass-deletes clients), but in normal use is rare.

### G.3 Transaction has no client match

`txId` references a transaction with `clientId = 999` (no such client). User taps the tx in Explore.
- Explore's `safeClients.find(c => c.id === tx.clientId)` returns undefined. The dot color falls back to `'#666'` (line 89).
- Tap → `router.push(...?id=999&...)`.
- client-detail's `clients.find(c => c.id === Number(999))` returns undefined.
- "Client not found." fallback shows.

The transaction is effectively unreachable for cleanup. This is an orphan-data state, not a user-facing flow, but worth knowing.

### G.4 Cleanup completes and filtered list is empty

After the redirect to `/explore?filter=X`:
- Explore re-renders with `filteredByParam` derived from current store state (line 35-41).
- If 0 results AND query empty, line 74-83 renders the per-filter empty state:
  - missing → "All receipts added — nothing missing."
  - untagged → "Everything is tagged."
  - flagged → "No transactions need review."
- The filter label "Showing: X" still shows at line 70 (because `filter` is still in the URL).
- Home's NEEDS ATTENTION row for that category will be hidden on next navigation to home (count = 0, filtered out at line 178-180).
- The user can manually pop back to home, or tap a different bottom tab, or pull down (no swipe-back configured here on Android).

There is no auto-exit from the empty filtered state. The user remains on Explore until they navigate.

## SECTION J — REAL-WORLD USAGE PATTERNS

These are the actual conditions Uncrumple cleanup runs in. Test plans built only against the happy path will miss most user complaints.

### J.1 The "end-of-week catch-up" pattern

A tradesperson logs transactions hastily during the day — dirty hands, in the truck between jobs, sun glare on the screen. They tap Add Transaction, type merchant + amount, skip the photo and tag. At the end of the week (Friday or Sunday) they run cleanup in one sitting.

Implications for testing:

- A typical cleanup session is 10–50 transactions in one sitting. Test endurance, not just single fixes.
- The user is doing the same fix repeatedly (Add Receipt × 30). If a redirect lags, the friction compounds across 50 fixes. Time the redirect from "tap Save" to "Explore filtered list rendered" — should feel under 300 ms on a mid-range Android.
- If a single redirect silently fails, the user may not notice for several items: they see a stale row at the bottom of the list, re-enter, find it's already fixed, exit. Confusing but not data-corrupting.
- At 50+ fixes per session, ANY per-fix Alert prompt is intolerable. Audit for unexpected modals: the "Don't forget to add a note!" Alert at line 234-244 of client-detail.tsx ONLY fires on Add (not Edit, not cleanup), so it's safe — but verify after any patch.
- Battery: a 30-minute cleanup session with camera launches drains 5–10% on most phones. Out of scope but inform users.

### J.2 The "I lost the receipt" pattern

The user opens a tx in cleanup, realizes they truly do not have the receipt (lost it, was paid in cash, never got one). They want a way to mark "no receipt available" and move on.

Current app has no such state. The only workaround is to add a placeholder note via the Edit form, exploiting the H.1 inconsistency: Home counts only `!receiptUri && !note`, so a noted-but-receiptless tx silently drops out of the Home banner. But the Explore filter (`!t.receiptUri`) still shows it.

Behavior matrix:

| Pre-state                                  | Home count | Explore missing list |
|--------------------------------------------|------------|----------------------|
| no receipt, no note                        | counts     | shows                |
| no receipt, note set                       | NOT counted| shows                |
| receipt set                                | not counted| not shown            |

Real-world consequence: the user wonders why Home says "0 missing" but Explore shows 5. They will not figure out why on their own. Document as a triage symptom; tracked as item H.1.

### J.3 The "wrong client" pattern

User is mid-cleanup, opens a tx, realizes it should belong to a different client.

Path:

- The detail sheet has NO Move-to-Client button. Move is only available from multi-select mode (line 532-538), which requires entering the list view, long-pressing the tx, then tapping Move.
- From cleanup the user must Close the sheet (closeTxDetail redirects to Explore). Then navigate to the client's detail screen via Home tile. Long-press, multi-select, Move.
- Friction: 5+ taps + a tab switch + a long press.
- After Move, the tx now belongs to a different client. If the user re-enters cleanup via the same filter, the tx will or will not be there depending on whether the destination client also has the issue. No data corruption, just navigational churn.

Test pattern: Tap into a cleanup tx, hit Close (no fix), navigate to client list, perform Move from there.

### J.4 The "interrupted by phone call" pattern

User taps Add Receipt, camera opens, phone rings. Most Android handlers background the camera while the call UI takes priority.

Behavior:

- expo-image-picker's `launchCameraAsync` is awaited at line 200 (Add Transaction) and line 310 (cleanup). The Promise typically resolves with `result.canceled === true` if the camera process is killed.
- If the OS kills the host RN app for memory pressure during the call, on relaunch the in-flight Promise is gone. The user lands on whatever Expo Router restores — typically the last tab. The cleanup state is lost.
- Variance: Samsung One UI handles call-during-camera differently from Pixel stock Android. Test on multiple OEMs.

Tests J4.1 through J4.3:

J4.1 — Decline call mid-camera: tap Add Receipt → simulate incoming call → decline → verify camera resumes.
J4.2 — Answer call mid-camera: tap Add Receipt → simulate call → answer → end → verify state.
J4.3 — App killed during call: tap Add Receipt → simulate call → answer → background app → use enough memory to force kill → end call → reopen Uncrumple → document behavior.

### J.5 The "two-thumb fast cleanup" pattern

User goes fast: tap row, tap Add Receipt, take photo, tap row, tap Add Receipt, take photo, ...

Failure modes specific to fast tapping:

- Double-tap Add Receipt: two `requestCameraPermissionsAsync()` calls fire. The second is queued by the OS or dropped. If both grant and both call `launchCameraAsync`, results may interleave. The second photo overwrites the first via line 313's `editTransaction`. Net: one of the photos is lost. The user may not realize.
- Tap Save while a previous Save is mid-commit: React batches state, but if user already navigated, the second commit writes to a tx in the previous screen's snapshot. Probably no harm — the tx id is captured before the await.
- Tap a list row while the previous detail sheet is mid-close-animation: usually the Modal's backdrop blocks underlying touches, but on slow devices touches can leak through. selectedTx gets overwritten before its closing animation completes. Last-write-wins; old state is lost.
- Tap Add Tags repeatedly: tag modal toggles open/close. State is consistent, just visually flickery.

Tests:

J5.1 — Spam-tap Add Receipt 3x quickly. Verify only one camera launches and only one photo persists.
J5.2 — Spam-tap Save in the tag picker. Verify no duplicate writes.

### J.6 The "I want to see all my Personal expenses" pattern

The Tax screen's `workTotal` excludes hardcoded `clientId !== 5` (line 36 of tax.tsx). This is the seeded "Personal" client's id.

Real-world breakage:

- If the user deletes the Personal client and re-adds one named "Personal", the new id is generated by the store's add logic. If addClient uses `Date.now()` (likely) the new id will not be 5. Then `workTotal` no longer excludes Personal expenses → tax savings overstate.
- If a different client coincidentally gets id=5 (extremely unlikely with Date.now ids, but possible if seed data is reset), that other client is silently excluded from tax.
- index.tsx's `outstandingTotal` excludes by name (`c.name !== 'Personal'`, line 170). So Home and Tax disagree on what "Personal" is.

Tests:

J6.1 — Rename Personal to "My Stuff". Verify Tax `workTotal` still excludes id=5 (correct, but cosmetically the Tax screen still treats it as "Personal" by exclusion logic). Verify Home `outstandingTotal` now INCLUDES it (since name is no longer "Personal").
J6.2 — Delete Personal entirely. Add a new client. Verify what id it gets and what each screen now excludes.

### J.7 The "I have 500 transactions" pattern

Performance considerations under realistic year-end data volume.

- Explore renders results via `.map(...)` (line 85), no FlatList virtualization. With 500 cleanup-eligible transactions, scroll is sluggish but not broken.
- client-detail's pending list (line 446) and invoiced groups (line 504) similarly use `.map`. A client with 200 pending transactions slows the entire screen.
- The detail sheet's tag picker (`topTags.filter(...)` at lines 672, 927) recomputes on every keystroke in the search field. Fine at 6 tags, sluggish at 100.
- `allTagsSorted` (line 124) sorts every render. With 50 tags × 1000 transactions, that's a 50000-op sort on every render. Should be memoized; isn't.

Tests:

J7.1 — Seed 1000 transactions, 200 untagged. Open Home. Tap "200 untagged". Time the Explore load.
J7.2 — Same dataset, open client-detail for a client with 200 pending. Time the screen.
J7.3 — Add a 51st tag. Open Add Transaction. Type in the merchant field. Watch for keystroke lag.

### J.8 The "month rollover" pattern

User is doing cleanup at 11:55 PM on the last day of the month. As they fix transactions, midnight passes.

Behavior:

- New transactions written get the new month's date string (`new Date().toLocaleDateString('en-US', { month: 'short', day: 'numeric' })` at line 226). No problem — they wanted the new date.
- But the Home header at line 187 reads the month/year ONCE at render time. If the screen is on display across midnight, the header is stale until the next render. Trivial.
- Tax screen: `year = new Date().getFullYear()` at line 21. Same staleness. Trivial.

### J.9 The "year rollover" pattern (more concerning)

December 31 → January 1.

- All existing transactions stored as `'Dec 31'` etc. — no year. New transactions stored as `'Jan 1'`. The string sort puts December LAST alphabetically (D > J). So a year-rollover dataset sorts ABCDFJ … (Apr, Aug, Dec, Feb, Jan, Jul, Jun, Mar, May, Nov, Oct, Sep). Garbage chronological order.
- Tax screen says "<currentYear> SUMMARY" but `yearTxs = transactions` (line 22) is unfiltered. So the 2026 SUMMARY view shows all 2025 + 2026 data combined.
- Cleanup is unaffected (filters don't depend on date).

Test J9.1: at year rollover, verify Tax screen behavior. Document for users that "<year> SUMMARY" is misleading.

### J.10 The "shared device" pattern

Couple shares a phone, both are tradespeople.

Not supported. The app has no user accounts. All data is one user. If shared, the cleanup pile mixes both users' transactions — they have no way to separate.

Out of scope for QA but document as a known limitation.

## SECTION K — FAILURE SCENARIOS

### K.1 AsyncStorage write failures

The store wraps every mutation with `AsyncStorage.setItem(...)` (e.g. line 166-167 of store.tsx). The Promise is fire-and-forget — not awaited, no `.catch`.

Failure consequences:

- In-memory state updates optimistically (the cleanup list reflects the fix immediately).
- Persistence silently fails (storage full, OS kills the write, AsyncStorage SQLite corruption).
- On next app launch, the load (line 73 of store.tsx) returns the last successfully-persisted state.
- The user's "fix" is gone. They re-enter cleanup, see the same tx still untagged, and assume the app is broken.

Test K1.1 — Simulate storage full:

1. Fill device storage to 99%+ (use test photos to consume space).
2. Open cleanup, tap Add Receipt, capture photo.
3. Verify in-session: tx is gone from list (in-memory write applied).
4. Force-close the app.
5. Reopen. Verify the tx state.
6. Document whether the receipt persisted or was lost.

Test K1.2 — Older Samsung AsyncStorage limit:

- On certain older Samsung (Android 9 era), AsyncStorage's SQLite backing has a 6 MB row limit. With 1000+ transactions and saved merchant strings, you can approach this.

- Test on a Galaxy J or A series phone if available.

- The detail sheet's `<Image source={{ uri }}>` renders a broken/empty box.
- The cleanup flow has no way to flag this — Missing Receipts filter does not re-detect missing files.

## SECTION Q — PERFORMANCE AND SCALE

### Q.1 Filter computation complexity

Every render of Explore (lines 35-41) runs ONE of three `safeTransactions.filter(...)` paths depending on `filter`. O(N).

Every render of Home (lines 174-176) runs THREE `.filter().length` operations in parallel. O(3N).

Every render of Tax (lines 24, 25, 27, 29-33, 35-37) runs five+ filter/reduce operations. O(N) each, ~6N total per render.

At N=10000 transactions, expect frame drops on every screen render.

Test Q1.1: seed 10000 transactions. Open Home. Time to first paint. Time to scroll-smooth.

### Q.2 Per-tx tag/client lookups

Every list row calls `tags.find(t => t.id === tid)` (line 467, 807, 911 of client-detail; line 86 of explore). O(M) per tag per render. For a tx with 5 tags rendered in a list of 100 txs: 500 lookups per render.

A Map index would be O(1) — not present in current code.

### Q.3 Sorted tags recomputation

`allTagsSorted` (line 124 of client-detail) recomputes every render of client-detail. The sort uses `transactions.filter(t => (t.tagIds || []).includes(tag.id)).length` — that's O(N×T) per render where N=transactions, T=tags.

At N=1000, T=20 → 20000 ops per render. OK on mid-range. Scales poorly.

### Q.4 Image loading

Receipt thumbnails (line 825) load full-resolution images for the detail sheet. expo-image-picker stores at quality 0.8 — typically 1-3 MB. RN Image component loads via native bitmap, decodes on the GPU.

Memory pressure: opening 50 different cleanup sheets in sequence (without restarting) causes the RN image cache to grow. On low-RAM devices (2GB), this can OOM.

Test Q4.1: on a 2GB Android device, open and close 50 unique cleanup transactions in 2 minutes. Monitor memory via `adb shell dumpsys meminfo <pkg>`.

### Q.5 Tag picker scroll performance

The tag modal (line 980) has `tagModalScroll` with `maxHeight: 300`. With 50 tags, the user scrolls. Each tag chip is a TouchableOpacity. Scroll is fine up to ~100 chips, sluggish beyond.

### Q.6 Search input keystroke lag

Add Transaction sheet's merchant input triggers `handleMerchantChange` (line 152) on every keystroke. It runs `savedMerchants.filter(m => m.toLowerCase().includes(lower))` synchronously. With 10000 saved merchants, each keystroke takes 50+ ms — visible lag.

H.11 Tax `workTotal` excludes hardcoded `clientId !== 5` (line 36). Index.tsx `outstandingTotal` excludes by `c.name !== 'Personal'` (line 170). Two screens use different exclusion mechanisms for the same intent. Renaming or deleting Personal breaks one or the other.

H.12 `detailMetaRow` (line 761) uses inline-color text via nested `<Text style={{ color: ... }}>`. CLAUDE.md says "All styles must use StyleSheet.create — no inline style objects except for one-off dynamic values". The colors here are dynamic per state, so this is borderline acceptable but inconsistent with the rest of the file's style discipline.

H.13 The detail sheet's `<Image source={{ uri: receiptUri }}>` (line 825) does not validate that `uri` is reachable. Broken URIs render as empty boxes. No error handler or fallback UI.

H.14 The receipt viewer (line 1041) uses ScrollView with `maximumZoomScale=3, minimumZoomScale=1` for pinch-to-zoom. **ScrollView zoom only works on iOS. On Android, pinch-to-zoom on a receipt photo does nothing.** User cannot enlarge to read fine print on a receipt photo on the primary target platform.

H.15 `addReceiptToTx` writes `editTransaction(txId, { receipt: true, receiptUri: uri })` (line 313) — sets BOTH the boolean and the URI consistently. The Remove Receipt flow (line 838) writes both back to false/null. Manually-added receipts in the cleanup flow are consistent. Only seed/legacy data has the receipt-vs-receiptUri mismatch.

H.16 The "+ Add Rule" link from a cleanup sheet (line 860-869):

```
const m = selectedTx?.merchant;
closeTxDetail();
router.push({ pathname: '/rules', params: { prefillMerchant: m } });
```

- `closeTxDetail` (line 259) sets selectedTx to null AND if source/returnFilter, calls `router.replace('/explore?filter=X')`.
- Then `router.push('/rules')` pushes /rules on top.
- Final stack: Home → Explore (filtered) → Rules.
- When the user taps back from Rules, they land on Explore (filtered), NOT on the cleanup sheet they were looking at.
- Intentional but worth knowing — the user may expect to return to the sheet to apply the rule's effect.

H.17 The cleanup buttons show in the detail sheet ONLY when `source === 'explore'` AND `returnFilter === 'untagged' || 'missing'`. If the user navigates via Home → client tile → into client-detail → opens a tx, the cleanup buttons are NOT shown even if the tx is missing tags or receipt. The detail-sheet cleanup affordance is contextual, not global. Users who don't enter via filtered Explore won't see it.

H.18 The `addReceiptToTx` redirect (line 316) requires `capturedTagIds.length > 0`. If the entry was via `?returnFilter=missing`, this is appropriate (the tx must also have tags to fully resolve). If a hypothetical future filter doesn't care about tags, this gating would be wrong. Currently no such filter exists; future-proofing concern only.

H.19 The detail sheet's view mode (lines 716-874) does NOT wrap content in a ScrollView. On smaller phones with content-rich transactions (long note + 8 tags + receipt thumbnail), the Close button can be clipped (P.1). The edit-mode form does have a ScrollView, but the more-frequent view mode does not.

H.20 Tag color generation in `handleCreateTag` (line 181) uses `TAG_CYCLE_COLORS[tags.length % 8]`. If a user creates and deletes tags, `tags.length` is the current count, so colors collide with existing tags. Two tags can share the same color silently.

H.21 Merchant suggestion logic at line 161 filters with `m.toLowerCase().includes(lower) && m.toLowerCase() !== lower`. The exclusion of exact-match means typing "Menards" exactly hides the saved "Menards" suggestion (because it's already typed). Correct in spirit but potentially confusing — typing the full name shows zero suggestions, then completing it hides what was just suggested.

H.22 The "Mark Reviewed" handler (line 770-779) does NOT confirm. A single tap commits and may auto-redirect. There is no undo. If a user taps it accidentally on a flagged tx, they can re-flag only via... actually, there is no in-app re-flag affordance. The flag was set by the rules engine on tx creation; once cleared, the only way to re-flag is to create a new matching rule and add a new tx (or manually via Edit if the form exposes the field — it doesn't, per lines 285-289).

H.23 The Tax-back interceptor (lines 16-23 of explore.tsx) installs only when `source === 'tax'`. After the user enters client-detail and returns via `router.replace('/explore?filter=X')`, the source param is dropped (line 88 hardcodes `source=explore`). Hardware back from the post-fix Explore no longer routes to Tax — the interceptor isn't installed. The Tax-back behavior is silently lost mid-flow.

H.24 The clone TEMP TEST button (lines 724-752) appends `__test_clone__` to the note. If the user adds notes after cloning, the marker stays in the middle of the note text. It's never re-stripped. Real user data ships with this marker if the button isn't removed before release.

H.25 Saved-merchants store grows unboundedly (`addMerchant` in store.tsx). Every Add Transaction adds the merchant to a saved list. There's no cleanup or deduplication beyond exact match. After 1000 transactions, the saved-merchants array is large and the keystroke filter (Q.6) lags.

H.26 The `safeTransactions = transactions.filter(t => t && typeof t === 'object')` in explore (line 30) and `safeTxs = transactions.filter(t => t && t.clientId != null && t.amount != null)` in client-detail (line 102) use different validation criteria. A tx with clientId=null is visible in Explore (and counted in cleanup) but invisible in the source client's pending list. Inconsistent.

H.27 Hardware back on a transaction with `selectedTx` set always calls `closeTxDetail` (line 81), which redirects unconditionally. There is no "back closes sheet without redirecting" affordance. Combined with H.4 (Close button also redirects unconditionally), the user has zero way to "close this sheet and stay on this page" mid-cleanup.

H.28 The cleanup `txMissingReceipt` and `txMissingTags` flags (lines 381-382) are computed from `selectedTx` state (the local snapshot), not from the live store. If something else mutates the tx mid-sheet (rules engine, future), the flags don't update until the user closes and reopens.

## SECTION T — TAB AND BACKGROUND INTERACTIONS

### T.1 Tab switch with cleanup sheet open

Bottom tabs are persistent. Pushed routes (client-detail) overlay tabs.

Behaviors to test (depends on root layout config — out of scope for these four files but documented for testing):

- If tab bar is hidden on pushed routes: not switchable. Cleanup state preserved.
- If tab bar is visible: tab switch may unmount client-detail or push a new tab on top.

Test T1.1: open cleanup → cleanup sheet visible → tap each bottom tab in turn → tap Home tab → verify whether the cleanup detail is still in the back stack.

### T.2 Tab badge / red dot for cleanup

None implemented. The Home banner (NEEDS ATTENTION) is the only cleanup indicator. The Search tab does not show a badge for filter results.

Future feature would add a numeric badge on the Search tab when filters have results. Document as not-built.

### T.3 Coming back to cleanup after "+ Add Rule"

Per H.16: cleanup → tap Add Rule → land on Rules screen → finish → press back → land on Explore (filtered), not on the cleanup sheet.

If the rules engine retroactively applies (per CLAUDE.md "Rules engine runs on every new transaction" — only NEW transactions), the existing cleanup pile is unaffected by the new rule. So the user's tap-Add-Rule action does NOT immediately reduce the current cleanup pile. They must add a new transaction matching the rule for it to fire.

Test T3.1: create a rule matching "Menards" via the cleanup → Add Rule path. Verify existing Menards transactions are NOT auto-tagged. Verify a NEW Menards transaction IS auto-tagged.

### T.4 Notification overlay during cleanup

System notifications (incoming SMS, calendar reminders) overlay as a banner. Tapping the banner backgrounds Uncrumple. Returning resumes the cleanup state.

Test T4.1: open cleanup → trigger a notification (e.g. send yourself an SMS) → ignore → resume cleanup. Verify state preserved.
Test T4.2: same → tap the notification (backgrounds the app) → return via Recents → resume. Verify state preserved.

### T.5 App update

Rare in-session: the user updates Uncrumple via Play Store while the app is open.

Behavior:

- Play Store kills the running app process.
- New version launches cold.
- AsyncStorage state survives (it's the same on-disk store).
- In-flight cleanup state lost.

If a future version changes the data model (e.g. adds receiptStatus field), the user's old data is loaded with the field undefined. All read code must handle this — currently no migration.

Test T5.1: hard to simulate. Document the data-model-migration risk as an architectural concern.
